## 4. 设计符合性检查（vs 架构设计.md / API-INTERFACE-DOC.md）

### 4.1 重大偏差

- **[B1] API 接口与文档严重不一致**
  文档（`API-INTERFACE-DOC.md`）定义：`/api/brain`、`/api/confirm`、`/api/dashboard`、`/api/verify-search`、`/api/webhook/register` 等。
  实际实现（`api/server.js`）：`/api/brain/pages`、`/api/drafts/:id/commit`、`/api/stats`、`/api/search`、`/api/generate-cases`，**且无 webhook 端点**；冲突解决文档为 `POST /api/conflicts/:id/resolve`，实际为 `PUT`。
- **[B2] 技术栈偏差**：架构设计写明 Python Flask 后端；实际为 Node.js + Express。
- **[B3] `agents/` 与 `config/` 目录为空**：架构要求 `agents/generator.json`、`agents/validator.json`（MCP 配置）及 `config/gbrain.config.ts`、`config/skills.config.ts`，均未创建。
- **[B4] 知识库空库**：`brain/quality-rules/`、`brain/defect-experience/` 目录为空，导致 `/api/stats` 的 `totalRules=0`、`totalDefects=0`，搜索/推荐无法利用质量规则与缺陷经验。
- **[B5] 已有 `docs/V1.0-acceptance-report.md` 声称「V1.0 验收通过、70/70 通过、功能完整性 100%」**，与上述实际偏差不符，结论过于乐观，建议以本报告为准。

### 4.2 正向符合

- 核心 PRD 能力（入库 / 冲突 / 质量门 / 搜索 / 图谱 / 审计）均有对应实现。
- Web UI 已实现（架构称「规划中 / 可选」）。
- 多格式上传、列表分页（迭代需求）已实现并验证。

## 5. 数据质量与一致性问题

- **[D1] 前端计数不一致**：草稿页标题「待入库草稿(111)」、导航 badge「草稿审核 8」、`/api/stats` 的 `pendingDrafts=0` 三处对不上（111 ≈ 103 原有 + 8 上传；stats 的 pending 统计与前端不同源）。
- **[D2] 冲突数不一致**：草稿页导航「冲突处理 2」、conflicts 页导航「冲突处理 0」、`/api/stats` 的 `conflictDrafts=2` 但 `totalConflicts=0`。
- **[D3] 质量规则 / 缺陷经验库为空**（见 B4）。

## 6. 结论与建议

1. 系统**核心闭环可运行**，但设计与实现存在系统性文档偏差，且缺少 MCP/配置层、关键知识库为空。
2. 建议：对齐 API 文档与实现、补齐 `agents/` 与 `config/`、填充质量规则与缺陷经验样本、修正前端计数逻辑。
3. 多项目知识库**隔离/共享**能力当前缺失，已作为独立改造项实施（见任务2），并同步更新 V1.0 设计文档（任务4）。
